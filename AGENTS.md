# AGENTS.md — WhatsApp AI Agent (Lusaka v1)

This document describes the project architecture, conventions, and guidelines for AI agents working on this codebase.

---

## What This Project Does

A multi-agent FastAPI backend that handles WhatsApp conversations via the **Meta WhatsApp Cloud API**. Incoming messages are classified, answered via RAG (FAISS + OpenAI), escalated to humans when needed, and synced with a stock management system (Mercado Fiel).

---

## Stack

| Layer | Technology |
|---|---|
| API | FastAPI + uvicorn |
| AI / LLM | OpenAI (`gpt-4o-mini` + `text-embedding-3-small`) |
| Vector store | FAISS (local, persisted in `data/faiss_index/`) |
| Database | PostgreSQL 15 (SQLAlchemy async) |
| Cache / Queue | Redis 7 |
| Containerization | Docker Compose |
| Tunnel (dev) | ngrok |
| WhatsApp | Meta Cloud API v21.0 |
| Stock API | Mercado Fiel (external HTTP) |

---

## Project Layout

```
agents/
  guardian_agent.py      # Classifies messages: VALID_QUERY, SPAM, SENSITIVE, STOCK_OPERATION, ESCALATION_REQUEST, GREETING
  rag_agent.py           # FAISS retrieval + OpenAI answer; logs confidence and sources
  handoff_agent.py       # Sends handoff template, calls pass_thread_control
  stock_agent.py         # Parses stock commands and calls Mercado Fiel API
api/
  webhooks.py            # POST /webhook/whatsapp — validates X-Hub-Signature-256, idempotency by message_id
services/
  whatsapp_service.py    # Async client for Meta Cloud API (send messages, mark read, pass_thread_control)
  template_service.py    # Fallback templates when outside the 24h messaging window
  mercadofiel_service.py # HTTP client for Mercado Fiel stock API
scripts/
  load_documents.py      # Indexes all Markdown in data/documents/ into FAISS + DB
  test_conversation.py   # Interactive CLI simulation (local stubs, no real WhatsApp)
data/
  documents/             # Markdown knowledge base files (products, schedules, policies)
  faiss_index/           # Persisted FAISS index (gitignored, generated by load_documents.py)
```

## Key Conventions

- **Idempotency**: Every incoming webhook is deduplicated by `message_id`. Never process the same message twice.
- **24h window enforcement**: If `OutsideMessagingWindowError` is raised, `TemplateService` sends an approved template (`session_expired` by default). Do not try to send a free-form message outside the window.
- **Signature validation**: `X-Hub-Signature-256` is verified in `api/webhooks.py`. Set `SKIP_WEBHOOK_SIGNATURE_VALIDATION=true` only for local testing.
- **Confidence logging**: RAG answers log `confidence` and `sources` as structured JSON via `structlog`. Confidence below 0.5 triggers handoff.
- **Template mapping**: `TEMPLATE_MAPPING` env var is a JSON dict mapping intent names to approved Meta template names.
- **Stock commands** (parsed by `stock_agent.py`):
  - `entrada <product_id> <qty>` — add stock
  - `salida <product_id> <qty>` — remove stock
  - `venta <product_id> <qty>` — record sale
  - `stock <product_id>` — query current stock

## Environment Variables

Copy `.env.example` to `.env` (or run `./setup.sh`). Key variables:

| Variable | Purpose |
|---|---|
| `OPENAI_API_KEY` | OpenAI API key |
| `WHATSAPP_PHONE_NUMBER_ID` | Meta phone number ID |
| `FACEBOOK_PAGE_ACCESS_TOKEN` | Meta permanent access token |
| `FACEBOOK_APP_SECRET` | Used to validate webhook signatures |
| `FACEBOOK_TARGET_APP_ID` | Handover target app (Page Inbox) |
| `WHATSAPP_WEBHOOK_VERIFY_TOKEN` | Token for Meta webhook verification |
| `SKIP_WEBHOOK_SIGNATURE_VALIDATION` | Set `true` for local dev |
| `WEBHOOK_BASE_URL` | Public URL (ngrok or production domain) |
| `MERCADO_FIEL_API_URL` | Mercado Fiel base URL |
| `MERCADO_FIEL_API_KEY` | Mercado Fiel auth token |
| `DATABASE_URL` | PostgreSQL connection string |
| `REDIS_URL` | Redis connection string |

## Local Dev Quickstart

```bash
# 1. Create .env with real credentials
./setup.sh

# 2. Start Docker, load documents, and expose via ngrok
./run.sh
```

After ngrok starts, copy the HTTPS URL and set it as `WEBHOOK_BASE_URL` in `.env`. Then configure the Meta webhook:
- URL: `https://<ngrok-url>/webhook/whatsapp`
- Verify token: value of `WHATSAPP_WEBHOOK_VERIFY_TOKEN`
- Subscribe to: `messages`, `message_status`

## Observability

Structured JSON logs via `structlog`. Key log events to search for:

| Event key | Meaning |
|---|---|
| `whatsapp_message_received` | Incoming webhook processed |
| `guardian_classification` | Message category assigned |
| `rag_answer` | RAG response with confidence + sources |
| `template_fallback_sent` | Outside 24h window — template sent |
| `whatsapp_pass_thread_control` | Handoff executed |

View live logs: `docker compose logs -f api`

## Agent Decision Flow

```
Incoming message
  └─ Guardian classifies
       ├─ SPAM / SENSITIVE / OFF_TOPIC → auto-reply, stop
       ├─ ESCALATION_REQUEST → Handoff agent
       ├─ STOCK_OPERATION → Stock agent → Mercado Fiel
       └─ VALID_QUERY / GREETING → RAG agent
            └─ confidence < 0.5 → Handoff agent
```

## What NOT to Change Without Care

- `api/webhooks.py` signature validation logic — breaks Meta webhook verification.
- `services/template_service.py` template names — must match approved templates in Meta Business Manager.
- FAISS index schema — if embeddings model changes, delete `data/faiss_index/` and re-run `load_documents.py`.
- `DATABASE_URL` / `REDIS_URL` defaults in `docker-compose.yml` — used by all services.
